# Cairn Architecture Review — 2026-07-06

Scope: full codebase audit of `src/` (~90K lines, 377 TS/Svelte files, 96+ API routes, 37 page-server files, 116 server modules), done via parallel deep-read passes over each dimension plus direct grep/read verification of the more surprising claims. Question to answer: **would a professional developer respect this codebase, or does it look thrown together after rapid parallel AI-assisted development?**

## Bottom line

**A professional would respect the bones and wince at one specific, very fixable thing.**

Security-sensitive code (auth, encryption, hardware-wallet signing, access control), TypeScript discipline, the database's connection/parameterization hygiene, and configuration/feature-flag management are all built the way a careful senior engineer would build them — centralized, consistently invoked, well-reasoned in their comments. Nobody would flag `secretKey.ts`, `auth.ts`, `webauthn.ts`, `psbt.ts`, or `settings.ts` in review.

What would make a professional raise an eyebrow: **single-sig wallets and multisig wallets are almost entirely parallel implementations** — separate route trees, separate API endpoints, separate hardware-signer components, separate PSBT/scan logic — with well over a thousand lines of duplicated or near-duplicated code, some of it duplicated *knowingly* (one file has a comment admitting it copies another's validation logic verbatim). This is the fingerprint of "single-sig was built first, multisig was built beside it instead of through it" — which is exactly what the git history shows. It's not sloppy code; it's *unconsolidated* code. It's also the one thing here that would genuinely slow a growing team down, because it means every wallet-feature bug gets fixed once and then has to be remembered and fixed again in the sibling tree.

The second-most-real issue is that the API layer, while consistently *authenticated* (there's a real shared `requireUser`/`requireAdmin`/`requireFeature` guard used everywhere), has no consistent *response contract* — six different shapes for a successful JSON response coexist across ~90 route files, which is a real papercut for anyone consuming this API from outside the app itself.

Everything else is refinement, not rescue.

## Detail by dimension

### 1. Project structure and directory layout — NEEDS WORK

The top-level shape (`src/lib/server`, `src/lib/components`, `src/lib/hw`, `src/lib/shared`, route groups `(app)`/`(auth)`) is exactly what an idiomatic SvelteKit project should look like, and the historical `vaults → multisig wallet` rename (commit `78a2864`) was executed properly — `hooks.server.ts` and `db.ts` carry deliberate, well-commented backward-compat redirects and a real table-rename migration, not a half-finished mess.

What isn't clean: `src/lib/server` has almost no internal structure. Alongside real subdirectories (`bitcoin/`, `chain/`, `channels/`, `electrum/`, `featureFlags/`, `wallets/`) sit 60+ loose top-level files, including at least six separate "multisig-something" files (`multisigExport.ts`, `multisigRoster.ts`, `multisigScan.ts`, `multisigShares.ts`, `multisigTransactions.ts`, `multisigBackupPdf.ts`, `multisigRegistrations.ts`) that were never grouped into a folder — every new feature added a same-level file instead of a module. And `src/lib/hw`'s per-route signer components tell the same story from the other direction: `wallets/[id]/send/_components/` and `wallets/multisig/[id]/send/_components/` are copy-paste sibling trees (`LedgerSigner.svelte` / `MultisigLedgerSigner.svelte`, etc.), not one component parameterized by wallet type.

### 2. Module separation (server vs. client, features vs. shared) — OK

Server/client separation holds up under direct inspection: `wallets/[id]/+page.svelte` and its multisig counterpart only call `fetch('/api/...')`, and every `$lib/server` reference found inside `.svelte` files is `import type` — erased at build time, zero runtime leakage. Some files (`NotificationPanel.svelte`, `activity/+page.svelte`) even carry explicit comments asserting the boundary. Business logic properly lives in `+page.server.ts`, not in components.

The gap: `src/lib/shared/` exists but holds only two files, so most engineers evidently didn't use it. The clearest symptom — `wallets/multisig/[id]/send/+page.svelte` imports `signerContract.ts` and `QrSigner.svelte` from the **single-sig** route's private `_components/` folder via a relative cross-route import, because no shared home was ever set up for them. The team clearly noticed the duplication in this one case and patched around it rather than promoting the file to `src/lib/shared`.

### 3. Hardcoded values vs. configurable constants — OK

The things that matter most are centralized: `MAX_FEE_RATE` lives once in `psbt.ts` and is correctly imported everywhere; Electrum/Esplora endpoints and their defaults live in one `settings.ts` `DEFAULTS` object, overridable via the DB and admin UI; Bitcoin network version-bytes (SLIP-132 tables, mainnet-only by explicit design) live in one `xpub.ts`; no committed secrets, API keys, or private keys were found anywhere in `src/` (only test fixtures with obviously-fake credentials).

What's duplicated instead of shared: `walletScan.ts` and `multisigScan.ts` each independently define an identical six-constant block (`GAP_LIMIT=20`, `BATCH_SIZE=20`, `HARD_CAP=400`, `TX_DETAIL_CAP=50`, `TX_FETCH_CONCURRENCY=8`, `CACHE_TTL_MS=60_000`) verbatim. Request-timeout constants are defined independently per network client (`esplora.ts` 12s, `electrum/client.ts` 15s, `webhook.ts` 10s) with no shared "network timeout" module. SMTP port `587` is hardcoded as a fallback in three separate locations. Coinbase maturity (`100` confirmations — correctly a consensus constant, not configurable) is duplicated between `psbt.ts` and `multisigPsbt.ts` instead of shared from one place. One genuinely unfinished item: `addressWatcher.ts` hardcodes `CONFIRM_THRESHOLD = 1` with its own comment admitting the intended configurable 1/3/6-confirmation design isn't wired up yet.

### 4. DRY violations and duplicated logic — NEEDS WORK

This is the headline finding, and it's structurally consistent rather than random copy-paste sloppiness — which is actually good news, because consistent duplication is easy to consolidate:

- **Verbatim-duplicated validation, self-documented as a duplicate.** `constructPsbt` (`src/lib/server/bitcoin/psbt.ts:237-258`) and `constructMultisigPsbt` (`src/lib/server/bitcoin/multisigPsbt.ts:258-279`) contain a near-identical 20-line recipient/fee-validation block (same checks, same error message strings). The multisig file even has a comment admitting it: `// ---- recipient / fee validation (same rules and messages as psbt.ts) ----`.
- **API handler duplication.** `/api/wallets/[id]/psbt/+server.ts` and `/api/wallets/multisig/[id]/psbt/+server.ts` share ~35-40 lines of near-verbatim request-shaping logic (recipient normalization, `onlyUtxos` regex filtering, identical feature-gate calls). The same parallel-tree pattern repeats across `receive`, `utxo-mass`, `history.csv`, `transactions`, `descriptor`, `coldcard`, `ledger-registration` — every wallet-type-specific resource exists twice.
- **Hardware-signer components are wholesale forks, not shared-and-parameterized.** `LedgerSigner.svelte` (339 lines) vs. `MultisigLedgerSigner.svelte` (666 lines), `BitboxSigner.svelte` (419) vs. `MultisigBitboxSigner.svelte` (538), `TrezorSigner.svelte` (336) vs. `MultisigTrezorSigner.svelte` (447), a Jade pair (361 vs. 525) — five hardware-device integrations duplicated across two wallet types, each pair a near-total rewrite rather than a shared base with a multisig-specific extension.
- **Scanning logic duplicated independently.** `walletScan.ts` and `multisigScan.ts` (505 lines) each hand-roll their own BIP-44 gap-limit discovery loop, scripthash batching, and 60-second caching — not just the constants (see §3), the algorithm itself.
- **Security-relevant query duplicated four times.** The multisig ownership check (`SELECT ... FROM multisigs WHERE id = ? AND user_id = ?` / `EXISTS(...)` wrapping it) is independently reimplemented in `wallets/multisig.ts` (twice), `multisigShares.ts`, and `multisigRegistrations.ts`, instead of one shared `assertOwnsMultisig()` helper — a DRY gap that matters more than most because it's an authorization check.
- **RBF/CPFP fee-bump service is a second full parallel implementation, and the code says so.** `transactions.ts` (1,066 lines) and `multisigTransactions.ts` (878 lines) share only one small helper; `bumpTransaction` (169 lines) vs. `bumpMultisigTransaction` (152 lines) are line-for-line parity on the RBF-signaling loop and BIP-125 rule-4 checks, and `buildCpfpDraft` (135 lines) vs. `buildMultisigCpfpDraft` (130 lines) are identical clamping/error-code/insert logic differing only in wallet-vs-vault wording. A comment at `multisigTransactions.ts:745-747` states outright that the fee math and guardrails are "identical" to single-sig — this was conscious copy-for-parity during the CPFP feature work, never extracted into a shared module. ~300 lines, plus another ~35-50 lines repeated across each of the four CPFP/bump API route handlers for HTTP status-code mapping.
- **The wallet-detail page components duplicate their CSS wholesale, not just their logic.** The `<style>` blocks in `wallets/[id]/+page.svelte` and `wallets/multisig/[id]/+page.svelte` are ~280-300 lines of near-verbatim CSS, copy-pasted rather than shared via a component or common stylesheet — the largest single duplicated block found in this audit.
- **Hardware-wallet drivers duplicate their own internal plumbing across devices, independent of the single-sig/multisig split.** Typed error classes have the identical shape defined four times, once per device (`ledger.ts`, `trezor.ts`, `bitbox02.ts`, `jade.ts`, ~44 lines total); SLIP-132 xpub normalization logic is near-identical across `ledger.ts`, `trezor.ts`, and `bitbox02.ts` (~100+ lines); a multisig key-path regex parser is copied verbatim between `ledger.ts` and `trezor.ts`. Notably, a code comment admits this is *deliberate* — each browser driver avoids importing a shared module "so this browser driver never imports server code" — so unlike the other findings here, this one is a known, accepted tradeoff rather than an oversight, and any fix needs a client-safe shared module rather than a server-side one.
- **Smaller, lower-signal duplication**: an identical `QR_OPTS` object copy-pasted between the two wallet-detail `+page.server.ts` files; an identical 14-line "coinbase UTXO + tip height" fetch block in both; 5 of 7-8 device-label key/value pairs duplicated between `wallets/labels.ts` and `wallets/multisig/labels.ts`; a wallet-name-by-id lookup query copy-pasted verbatim (with the table name templated) between `addressWatcher.ts` and `backups.ts`.
- **Confirmed non-issue, worth naming so it isn't re-flagged later**: the empty `src/routes/(app)/vaults` and `src/routes/api/vaults` directories are untracked leftovers from the `vault → multisig` rename (commit `78a2864`), not a live parallel feature — safe to delete, cosmetic only.

Rough scope: comfortably **1,500-2,000 lines** of duplicated or near-duplicated logic, concentrated in the wallet feature area but spanning route, service, hardware-driver, and component/CSS layers simultaneously — this is larger than initially estimated once the CPFP/RBF service layer and the CSS blocks are counted alongside the API/PSBT/scan/component duplication above.

### 5. TypeScript type safety — GOOD

`tsconfig.json` has `"strict": true` with no loosened flags. `: any` appears only ~11 times in the entire codebase, concentrated almost entirely in hardware-wallet transport code where vendor SDKs are genuinely untyped (`ledger.ts`, `jade.ts`, `bitbox02.ts`) — a defensible, contained escape hatch. `@ts-ignore`/`@ts-expect-error` appear **zero** times anywhere in `src/`, real discipline. `svelte-check` runs clean and fast (1,628 files, under 2 seconds) with only 2 errors, both legitimate possibly-undefined guards sitting in the currently-uncommitted diff on this branch — not legacy debt. Domain modeling is genuinely coherent where done: `wallets/multisig.ts` defines real discriminated unions (`MultisigScriptType`, `MultisigKeyCategory`, `MultisigDeviceType`) with comments explaining *why* fields are optional.

The one soft spot: 51 occurrences of `as unknown as` double-casting, worst concentrated in `deviceRead.ts` (both wallet-type variants), `wallets.ts`, `contacts.ts`, and `addressWatcher.ts` — a signal that DB-row-to-domain-object mapping isn't properly typed in a few places and gets papered over with a cast rather than fixed. Also notable: single-sig wallet logic sits in one flat `wallets.ts` file while multisig got refactored into `wallets/multisig.ts`, with no shared base type unifying the two row shapes — they evolved independently rather than from a common hierarchy, echoing the DRY finding above.

### 6. Error handling consistency — OK

Logging discipline is genuinely good: **zero** raw `console.error`/`console.log` calls anywhere in `src/` — everything goes through the structured pino logger, and spot-checked catch sites consistently log context (`{err, walletId, userId}`) before responding.

Where it's inconsistent: there is no centralized error-response helper (`apiError`/`handleApiError` — zero hits), and two genuinely different idioms coexist for the same problem. `+server.ts` API routes mostly hand-roll `json({ error: '...' }, { status: N })` (237 occurrences); `+page.server.ts` load functions mostly use SvelteKit's `error()` throw helper (44 occurrences). These are split cleanly along route-type lines but there's no written rule tying them together, and the error *body* shape varies even within the `json()` convention (`{error}` vs. `{error, code}`). There's also a real, if narrower-than-a-prior-audit-found, tail of silent failure: roughly 25 catch blocks in `src/lib/server/**` catch an error and just `return`/`return null`/`return []` with no log call and no propagation — invisible to both the caller and the logs. This is the same "recurring theme" a prior internal tech-debt audit flagged; it's been reduced, not eliminated.

### 7. Database access patterns — OK

This is more disciplined than the "90K lines of rapid AI-assisted development" label would predict. There is exactly one `DatabaseSync` connection (`node:sqlite`, not an ORM) opened in `src/lib/server/db.ts`, imported everywhere via ES-module singleton caching — no connection sprawl. Every value-carrying query uses `?` placeholders; a thorough scan found no SQL-injection surface anywhere (the handful of template-literal SQL strings only interpolate hardcoded table-name constants chosen via a type-safe ternary, never user input). Real transaction discipline exists around the highest-stakes multi-table writes: the full backup-restore routine, atomic account+first-passkey creation, and multi-row preference updates are all correctly wrapped in `BEGIN`/`COMMIT`/`ROLLBACK`.

The genuine gaps: there is no migration tooling or versioning whatsoever — the entire schema and its ~15+ incremental `ALTER TABLE` changes live in one 865-line `db.ts`, re-evaluated via `PRAGMA table_info` introspection on every process boot, with no way to know what schema version a given `cairn.db` file is at other than "whichever ALTERs have already run." Three newer background-scan modules (`keyHealth.ts`, `backupHealth.ts`, `multisigRoster.ts`) have real, unguarded N+1 loops — one un-batched write per stale key/wallet/roster member, not wrapped in a transaction, and compounded by `notify()` itself doing 2+ writes per call. One genuine concurrency gap: `multisigTransactions.ts`'s `attachMultisigSignature` does a read-combine-write sequence with no transaction boundary — two concurrent signature submissions could race and one signer's contribution could get silently clobbered. And the ownership-check query duplication noted in §4 is a DB-layer symptom as much as a DRY one.

### 8. API route design consistency — NEEDS WORK

The strongest part of this layer is real and worth stating plainly: `src/lib/server/api.ts` provides `requireUser`, `requireAdmin`, and `requireFeature`, and all ~86-96 `+server.ts` files import from it. There is **no route found with a missing or forgotten auth check** — the only unauthenticated route (`/api/health`) is deliberately and explicitly documented as public. `hooks.server.ts` populates `locals.user`/`locals.flags` centrally. This is a real, non-trivial strength that a lot of hand-built AI-generated code gets wrong.

What isn't consistent is everything downstream of the auth check. At least **six different conventions for a successful JSON response** coexist across the route tree: wrapped single-key (`{wallet}`), wrapped multi-key flat objects, bare service-return passthrough with no wrapper at all, `{ok: true}` acknowledgments, raw non-JSON `Response` objects for file downloads/SSE streams, and — worst of the group — at least one route (`.../keys/[keyId]/verified`) that encodes a semantic failure as a 200-status body field rather than an HTTP status code. Error responses are more consistent (`{error, code?}` dominates), but the shared guards in `api.ts` itself throw via SvelteKit's `error()`, producing a third, structurally different shape from the app's own convention. There is **zero API versioning** — no `/api/v1/` anywhere — so any breaking response-shape change requires an atomic, coordinated deploy across all ~65 client call sites with no compatibility window. Naming mixes REST-resource style and RPC-verb style freely within the same feature area (`transactions/[txId]` sits next to `transactions/cpfp`), and feature-flag gating isn't applied by a consistent risk tier (money-movement routes are gated, but wallet/account deletion routes are not, for no documented reason).

### 9. Svelte component architecture — NEEDS WORK

Svelte 5 runes usage is completely consistent — zero files use legacy `export let`, 75 files use `$props()`, 51 use `$state()` — which is a real signal of discipline holding across many sessions rather than drift. Data-fetching is properly pushed to `+page.server.ts` everywhere checked; components don't do their own business logic.

The problem is size and duplication, not idiom. Four page components run 1,500-2,700 lines (`wallets/multisig/new/+page.svelte`: 2,663; `wallets/[id]/send/+page.svelte`: 2,380; `wallets/multisig/[id]/send/+page.svelte`: 2,373; `wallets/new/+page.svelte`: 1,528), mixing markup, hardware-device I/O, fetch calls, and validation in one file each — these four alone are roughly a fifth of the entire `.svelte` line count. There is no shared UI kit: `src/lib/components` has no `Button.svelte`, `Modal.svelte`, or `Input.svelte`, so 36 different route files hand-roll `<button class="...">` a combined 118 times. And as covered in §4, the hardware-signer components are wholesale duplicated per wallet type rather than parameterized. One bright spot: `src/lib/components/portfolio` (`AllocationBar`, `BalanceChart`, `RecentActivity`, `Sparkline`) is a genuinely well-factored, appropriately-sized component set — proof the team can do this well when it isn't racing to ship a second wallet type beside the first.

### 10. Test structure and organization — OK

64 test files, all consistently colocated (`foo.ts` + `foo.test.ts`), running clean: `npx vitest run` completes in ~8 seconds with **1,088 tests, 0 failures**. Coverage depth where it exists is genuinely strong and includes negative/error-path cases, not just happy paths — PSBT construction (single-sig and multisig), CPFP fee math and stuck-tx routing, auth (registration, passkeys, password, admin lockout guards), encryption round-trips, and all five hardware-wallet drivers at the encoding/parsing layer are well covered, several with references to real BIP test vectors and internal bug-tracking IDs in their descriptions.

The gap is concentrated and important: of 133 route files (`+page.server.ts` + `+server.ts`), only about 2 have any colocated test. The actual send/sign/broadcast HTTP paths — the exact code paths this branch's uncommitted diff touches — have zero automated coverage; only the underlying `lib/server` logic they call is tested. A regression test for a past access-control bug (`multisigAccess.test.ts`) explicitly documents that the earlier bug wasn't in the gate logic itself, it was that nothing had wired the gate into the route — and that class of bug still has no route-level test to catch a recurrence. There's also a `.hw-emu-test/` hardware-emulation harness at the repo root that's a real, useful integration-testing effort but sits entirely outside the automated `vitest` run.

### 11. Scalability for team development — NEEDS WORK

Several feature areas — notifications, admin, explorer, auth — are cleanly modularized and could absorb multiple engineers today without collision. Wallet features are the exception, and it's exactly because of §4: a bug fix in fee estimation, coin selection, or a hardware-signer flow has to be found and applied twice (once per wallet type) or it silently diverges between single-sig and multisig, which is precisely the kind of thing that causes production bugs once a team's shared memory of "where the logic actually lives" gets thin. The four 1,500-2,700-line page files compound this: two engineers assigned to "fix a bug in the send flow" and "add a feature to the send flow" will conflict on the same file constantly. Fixing the wallet-type duplication is what buys this project room to add more than one or two concurrent contributors.

### 12. Code documentation — OK

No CLAUDE.md exists despite this being an AI-assisted codebase — a surprising omission given the scale of work. README.md, by contrast, is substantial (140 lines) and accurate against spot-checked code. File-header comments in `src/lib/server` are unusually good: 93 of 116 `.ts` files have a `/**`-style header, and the prose is substantive rather than boilerplate — `addressWatcher.ts` explains its algorithm and failure semantics and cross-references a planning doc; `backup.ts` documents its security invariants explicitly. That quality doesn't carry down to individual functions, though — only a handful of the 330+ exported functions/consts across `lib/server` have their own doc comment; most rely entirely on the file header for context. Route server files are comment-light by comparison (`wallets/[id]/+page.server.ts`: 11 comment lines in 144; the multisig equivalent: 19 in 180) — functional but sparse. Almost no stale TODO/FIXME debt was found; the one real TODO in non-test code reads as a live, intentional flag rather than an abandoned comment.

### 13. Security pattern isolation — GOOD

No Bitcoin private key, xprv, seed, or mnemonic material exists anywhere in `src/lib/server` or `src/lib/hw` — Cairn is architecturally xpub-only/watch-only, and this is stated explicitly in code comments in multiple places, not just true by omission. All five hardware-wallet drivers were read in full and confirmed to only ever handle fingerprints, xpubs, and PSBTs — never key material, which is exactly what a proper HW-wallet integration should guarantee. Encryption is centralized in one module (`secretKey.ts`: AES-256-GCM with HKDF-SHA256 key derivation and domain separation, fresh IV per encryption, GCM auth-tag verification on decrypt) and reused across three genuinely different secret domains (per-user SMTP passwords, the Nostr instance signing key, with a documented migration path from an earlier plaintext-storage bug) rather than reimplemented per feature. The one other AES-GCM implementation in the codebase (`backup.ts`, for passphrase-protected backup files) is a deliberate, documented, separate choice — not accidental duplication — because it needs a human-passphrase KDF that the instance key doesn't. Password hashing (scrypt with `timingSafeEqual` comparison) and WebAuthn ceremony logic are each isolated in their own single module. The multisig owner/viewer/cosigner access gate lives in exactly two functions (`getViewableMultisig`, `getSignableMultisig`) called consistently from roughly 15 route files, rather than being reimplemented per route — and a regression test documents a past bug where the fix was specifically to make sure every route actually calls these functions, evidence the team has already been burned by the alternative once. A targeted check of logging call sites near password/secret/PSBT-adjacent code found no instance of a real secret value being logged — only metadata (user IDs, error objects, status codes).

### 14. Configuration management — GOOD

`process.env` access is genuinely concentrated — only a couple of files reference it directly outside test infrastructure; production code reads everything through `settings.ts`'s DB-backed `getInstanceSettings()`/`getChainConfig()`. That function cleanly separates a "public mode" (forces trusted, TLS-verified defaults) from a "custom mode" — a deliberate security-conscious split, not an afterthought. Feature flags are a strong, consistent pattern: the registry uses a TypeScript literal-type trick so a flag cannot type-check if it's shipped in a pre-disabled state, a genuinely clever fail-safe rather than a conventional boolean-config approach. There's direct evidence the team self-corrected toward consistency here: a comment in `notifyConfig.ts` documents that it replaced four independently hand-rolled secret-redaction implementations (across the settings page, channel API, admin API, and admin loader) with one shared pair of helpers — exactly the kind of consolidation the wallet-duplication problem (§4) still needs. The one soft spot: startup tasks are wrapped in try/catch that only logs by explicit design ("never throws here"), which means a misconfigured Electrum host or a broken background watcher fails invisibly into the logs rather than blocking startup — a reasonable tradeoff for a self-hosted single-instance app, but worth knowing about.

## What to refactor first (highest ROI, in order)

1. **Collapse the single-sig/multisig wallet duplication.** This is the one finding that shows up independently in structure (#1), DRY (#4), type modeling (#5), components (#9), and team scalability (#11) — fixing it clears more of this review than anything else. Concretely, in rough priority order: extract a client-safe `hw/common.ts` for the hardware-driver error classes and xpub-normalization logic (lowest risk, ~250 lines reclaimed, and the one piece of this cluster that doesn't touch server code); extract one parameterized PSBT-validation function used by both `constructPsbt` and `constructMultisigPsbt`; unify `transactions.ts`/`multisigTransactions.ts` behind one fee-bump/CPFP service parameterized by wallet type (~300 lines, but this is money-moving code, so it needs solid test coverage — see #3 — before touching it); build one parameterized API-handler wrapper for the `psbt`/`receive`/`utxo-mass`/`transactions`/`descriptor` resource pairs; extract one `GapLimitScanner` with a pluggable address-derivation strategy instead of two hand-rolled scan implementations; share the wallet-detail pages' CSS instead of duplicating the `<style>` block; and turn the five paired hardware-signer components into one component per device with a `multisig?: MultisigContext` prop instead of two components per device.
2. **Give the API layer one response contract.** Pick one success-envelope convention and one error-envelope convention, write them down, and migrate the ~90 route files onto it (this is mechanical, low-risk, high-value work — good to do as a follow-on once #1 has already touched most of these files). Fix the one 200-status-encodes-failure route while at it.
3. **Add route-layer test coverage for the money-moving flows.** Backend PSBT/CPFP/auth logic is well tested; the actual send/sign/broadcast HTTP paths that a real user (or attacker) hits have effectively zero coverage, and this codebase has already been bitten once by a route-wiring bug that only route-level tests would catch. Do this incrementally as part of #1's refactor, since you'll be touching every one of these routes anyway.
4. **Close the two DB-layer correctness gaps found in this review.** Wrap `attachMultisigSignature`'s read-combine-write in a transaction to remove the concurrent-signing race, and wrap the three N+1 notification-scan loops (`keyHealth.ts`, `backupHealth.ts`, `multisigRoster.ts`) in transactions and batch their writes — small, contained fixes with real correctness value.
5. **Break up the four 1,500-2,700-line page components** into sub-components (device-signing steps, QR flows, fee/coin-selection panels) — do this alongside #1, since the wallet-type consolidation touches these same files anyway, making the extraction close to free.

## What can wait

- Formal migration tooling for the DB schema — the current hand-guarded `ALTER TABLE` approach is unusually well-commented and works fine at this project's current scale; worth revisiting only if the schema keeps growing or a second environment/rollback story becomes necessary.
- The small duplicated constants (SMTP port fallback, per-client request timeouts, coinbase maturity literal, scan-cache tuning knobs) — real but low-risk; a `constants.ts` sweep is a good low-stakes first PR for a new contributor.
- Per-function JSDoc coverage and a top-level CLAUDE.md / architecture reference doc — write the architecture doc *after* #1 lands, so it describes the consolidated shape rather than going stale immediately.
- API naming quirks (`.csv` in the path, verb-style `/stateless/*` routes, mixed REST/RPC naming) — isolated and not spreading.
- A shared reusable UI kit (`Button`/`Modal`/`Input`) — real technical debt, but lower urgency than the wallet-logic duplication since it's cosmetic/maintenance cost rather than a correctness or collision risk.

## What's already good enough to leave alone

Security and cryptography architecture (encryption, password/session handling, WebAuthn, hardware-wallet key isolation, the collaborative-custody access gate), TypeScript strictness and domain typing, the database connection/parameterization/critical-path-transaction discipline, structured logging conventions, the feature-flag system's fail-safe design, and the env-var/settings split are all at a standard a professional reviewer would sign off on without comment. Don't refactor these to satisfy an abstract sense of "more architecture" — they are not the problem, and touching them risks introducing the very inconsistency this review is flagging elsewhere.
