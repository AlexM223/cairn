### 2a. Fable's recommendation for the hooks.server.ts async-init fix (USE THIS)
**Premise correction from Fable:** with the current `instanceModeMigration.ts` formula (`users > 1 || shares>0 || invites>0 || contacts>0`), a clean first boot decides **'solo' whether it counts 0 OR 1 users** — so the misordering does **not** actually flip solo/team on a normal fresh install. The real defects being fixed are: (a) the dead `try/catch` → bootstrap failure becomes a generic `unhandledRejection`, losing the specific error message; (b) the documented "runs after bootstrap" invariant is silently false (latent trap if the mode formula ever changes); (c) a **new first-request race**: a signup committing during the ~100ms hash window can create a SECOND admin (human + the `admin@cairn.local` placeholder), and `migrateInstanceMode` then counting 2 users decides **'team'** — the actual way mode can go wrong. Near-theoretical (a human can't submit in 100ms) but real.

**Recommended pattern (Fable, decisive):** convert the ENTIRE top-level init block into one ordered `async function init()` that preserves the exact current order and per-step `try/catch` bodies, kick it off once at module scope, and **gate `handle` on its promise**:
```ts
async function init(): Promise<void> {
  try { await bootstrapAdminFromEnv(); } catch (e) { errLog.error({ err: e }, 'admin bootstrap from env failed'); }
  try { migrateInstanceMode(); } catch (e) { errLog.error({ err: e }, 'instance mode migration failed'); }
  try { migratePlaintextSecretsAtRest(); } catch (e) { ... }
  try { ensureDefaultAgreementVersion(); } catch (e) { ... }
  // ...ALL start*Watcher() calls, identical bodies, identical order...
}
const initReady = init();               // module scope — watchers must start even with no request; can't reject (every step catches)
export const handle = async ({ event, resolve }) => { await initReady; /* ...rest unchanged... */ };
```
Rationale: (i) restores exact pre-48hm ordering; (ii) each step keeps its own catch → `initReady` never rejects → no unhandledRejection, no `.catch` needed; (iii) `handle` is still defined synchronously (export not blocked); after init settles, `await initReady` is a single resolved-promise microtask/request (negligible, incl. assets); (iv) **zero top-level await, zero build-shape change** — the prod-build landmine evaporates. Gating `handle` reproduces the OLD observable guarantee (server.mjs swapped in the handler only after the sync module eval finished, so no request ever hit SvelteKit pre-bootstrap) while keeping the event loop free — strictly better than both old (event-loop-blocked) and ungated-async (race). No `globalThis` guard needed on `initReady` (steps are idempotent; dev-mode re-eval is fine).
- **Rejected:** native TLA (re-introduces the backed-out shape, extends server.mjs's 503 "starting" window, no upside); pair-only IIFE (introduces a *new* reordering of the later steps and doesn't close the race).
- **Ordering deps to preserve inside `init()`:** `migrateInstanceMode` after `bootstrapAdminFromEnv`; `migratePlaintextSecretsAtRest` BEFORE `startNotificationQueueWorker` (documented at hooks.server.ts:97-99). Everything else is independent, but move the whole block in anyway (all sync, mechanical wrap+indent, preserves both invariants structurally).
